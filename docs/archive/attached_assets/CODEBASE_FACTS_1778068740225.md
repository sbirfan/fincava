## Auth System — CRITICAL (Updated)

**JWT-based tokens, cookie-primary with Bearer fallback.**

```typescript
// requireAuth reads:
// 1. Cookie: req.cookies.fincava_auth (HTTP-only cookie — primary)
// 2. Header: Authorization: Bearer <token> (fallback for programmatic clients)
// Both are JWT tokens signed with JWT_SECRET

// requireAuth sets on req:
(req as any).userId   // number — the logged-in user's ID
(req as any).userRole // string — "BUYER" | "SUPPLIER" | "ADMIN"

// NOT req.user — there is no req.user object
// NOT req.supplierId — does not exist
```

**Existing middleware:**
- `requireAuth` — in `lib/auth.ts` — verifies JWT from cookie or Bearer token
- `requireVerifiedEmail` — in `lib/auth.ts` — must call AFTER requireAuth; 403 if email not verified
- `requireAdmin` — in `middleware/admin.ts` — checks `(req as any).userRole === 'ADMIN'`
- `adminOnly` — `[requireAuth, requireAdmin]` as const — use this for admin routes
- `requireRole(role)` — async factory, rarely used
- **`requireSupplier` — DOES NOT EXIST. Not needed for Phase I (officer-mediated).**

**Additional exports in `lib/auth.ts` (do not accidentally overwrite on edit):**
- `getUserWithProfile(userId)` — fetches user + profile rows together
- `hashPassword(password)` — bcrypt, 12 rounds
- `verifyPassword(password, storedHash)` — handles both bcrypt and legacy SHA-256 hashes
- `generateToken(userId)` / `verifyToken(token)` — JWT helpers

## Supplier Identity — CRITICAL DESIGN GAP (Unchanged)

**There are TWO supplier populations in this codebase. This is UNRESOLVED.**

| Population | Table | Identity | Auth |
|------------|-------|----------|------|
| Farm suppliers | `suppliersTable` | `whatsappNumber` | **No user account linked** |
| Marketplace suppliers | `usersTable` (role=SUPPLIER) | email + password | JWT via `requireAuth` |

`suppliersTable` has **no `userId` FK**. Farm suppliers onboarded via WhatsApp have no corresponding row in `usersTable`.

**MVP Decision (confirmed from ops/mvp_validation_plan.md):** Supplier self-service is Phase 2. MVP = admin/officer-mediated. CC-1D compliance flows are officer-facing, not supplier self-service. No supplier auth needed for Phase I compliance UI.

### Two-track model (actual implementation)
Single `supplierType` enum on `suppliersTable` — NOT separate entity tables:
- Track 1 → `supplierType IN ('FARMER', 'COOPERATIVE')` — graduation pipeline applies
- Track 2 → `supplierType = 'EXPORTER'` — product verification only
- Additional values present: `PROCESSOR`, `DISTRIBUTOR`, `OTHER`
- `OTHER` uses a separate `customSupplierType` varchar column for freeform admin entry
- **Any CASE statement or WHERE clause on supplierType must handle all 6 values.**

⚠️ NOTE: The git repo at `Archive2/Recalibrate/FinCava-Hub` only shows 3 enum values (FARMER, COOPERATIVE, EXPORTER). The live Replit codebase has 6. Trust the live Replit version — git repo is behind.

## Route Registration Pattern — CRITICAL

**All routers in `routes/index.ts` are mounted with NO path prefix.** Each router file embeds its full paths (including `/api/`).

```typescript
// routes/index.ts pattern (confirmed):
router.use(suppliersRouter);    // NOT router.use('/suppliers', suppliersRouter)
router.use(adminRouter);        // NOT router.use('/admin', adminRouter)
router.use(officersRouter);     // etc.

// Consequence for new routes: embed FULL path in router file, then:
router.use(adminComplianceRouter);  // ← correct CC-1D registration
// NOT: router.use('/admin/compliance', adminComplianceRouter)  ← WRONG
```

Routes within each file use full paths:
- `router.get('/api/admin/compliance/dian-rut/:supplierId', ...)`
- `router.post('/api/admin/compliance/dian-rut/:supplierId/intake', ...)`

Exception: `officers.ts` uses `pool.query` directly (not drizzle) — single registration route, only exception to drizzle pattern in codebase.

## File Uploads — INFRASTRUCTURE EXISTS

**A complete object storage system is already live.**

```
artifacts/api-server/src/lib/objectStorage.ts   ← ObjectStorageService class
artifacts/api-server/src/routes/storage.ts       ← Live endpoints:
  POST /api/storage/uploads/request-url          ← generates presigned upload URL
  GET  /api/storage/objects/public/*             ← serves public objects
  GET  /api/storage/objects/private/*            ← serves private objects (ACL checked)
```

**Upload pattern for CC-1D compliance documents:**
1. Client calls `POST /api/storage/uploads/request-url` → receives presigned URL
2. Client uploads file directly to presigned URL (no multer needed on the API)
3. Client POSTs the resulting `fileUrl` (object path) to the compliance route
4. Compliance route stores `fileUrl` in `complianceDocumentsV2.fileUrl`

Key function: `ObjectStorageService.getObjectEntityUploadURL()`

Do NOT add multer or any server-side upload middleware — the presigned URL pattern handles it entirely at the client layer.
