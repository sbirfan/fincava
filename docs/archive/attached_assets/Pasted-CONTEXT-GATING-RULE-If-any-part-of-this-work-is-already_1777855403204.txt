FEATURE 3a — Admin Profile Tab

Tab placement: Add "Profile" as the 4th tab in the buyer detail drawer (after Overview / Matches / Gaps, before Activity).

Tab content — two parts:

PART 1: P2 Profile display + admin override
Display all 23 P2 fields from buyer_profiles in four collapsible section cards (same section structure as /dashboard/profile: Company Profile / Product Interests / Volume & Pricing / Quality & Values).

For each field: show current value (or "—" if null) alongside an editable input.
Admin can edit any field. Save button per section (explicit, not debounced — admin edits are intentional overrides).

API for admin read: GET /api/admin/buyers/:id/onboarding (new endpoint)
  Returns all 23 P2 fields + p2CompletionPct + p2SectionsDone + p2ApprovalStatus
  Auth: requireAdmin

API for admin save: PATCH /api/admin/buyers/:id/onboarding (new endpoint)
  Accepts any subset of the 23 P2 fields
  Does NOT recalculate p2CompletionPct — admin overrides are raw field updates only
  Returns updated fields
  Auth: requireAdmin

PART 2: Approval workflow controls (below the section cards)
Show current approval status with a badge (see F3b for status definitions).
Three action buttons (shown based on current status — see logic below):
  "Approve Profile" → sets p2ApprovalStatus to APPROVED
  "Request Revisions" → sets p2ApprovalStatus to REVISION_REQUESTED; reveals a text input for the revision note (plain text, max 500 chars); on submit, saves note and fires buyerRevisionRequestedEmail
  "Needs Attention" → sets p2ApprovalStatus to NEEDS_ATTENTION (for flagged issues without a buyer-facing note)

Button visibility rules:
  Status PENDING_REVIEW: show all three buttons
  Status APPROVED: show "Request Revisions" and "Needs Attention" only
  Status REVISION_REQUESTED: show "Approve Profile" and "Needs Attention" only
  Status NEEDS_ATTENTION: show "Approve Profile" and "Request Revisions" only

All three actions call PATCH /api/admin/buyers/:id/approval (new endpoint):
  Body: { status, revisionNote? }
  Auth: requireAdmin
  On REVISION_REQUESTED: fire buyerRevisionRequestedEmail (fire-and-forget)
  Returns updated p2ApprovalStatus + revisionNote

FEATURE 3b — Approval Workflow

Schema migration:
Add two columns to buyer_profiles:
  p2_approval_status text DEFAULT 'PENDING_REVIEW' NOT NULL
  p2_revision_note text (nullable)

Use direct DDL (established pattern — do NOT run Drizzle omnibus migration):
  ALTER TABLE buyer_profiles ADD COLUMN IF NOT EXISTS p2_approval_status text NOT NULL DEFAULT 'PENDING_REVIEW';
  ALTER TABLE buyer_profiles ADD COLUMN IF NOT EXISTS p2_revision_note text;
Verify both via information_schema.columns after applying.

Status values: PENDING_REVIEW | APPROVED | REVISION_REQUESTED | NEEDS_ATTENTION
(No REJECTED — account-level decisions are separate from profile approval)

Auto-transition rule: when PATCH /api/buyer/onboarding sets p2CompletionPct to 100 for the first time (previous value < 100, new value = 100), automatically set p2ApprovalStatus to PENDING_REVIEW if current status is null or PENDING_REVIEW. Do not change status if it is already APPROVED, REVISION_REQUESTED, or NEEDS_ATTENTION.

New email template — buyerRevisionRequestedEmail(opts):
  opts: { buyerName, buyerEmail, revisionNote, profileUrl }
  Subject: "Action needed: your Fincava sourcing profile needs updates"
  Body: greeting, explain the team has reviewed their profile and needs some updates, show the revision note in a highlighted box, CTA button "Update My Profile" → profileUrl
  Follow existing baseTemplate/esc pattern in email.ts
  Fire-and-forget on PATCH /api/admin/buyers/:id/approval when status = REVISION_REQUESTED

Buyer-facing status badge:
Add to /dashboard/profile.tsx at the top of the page, below the heading:
  PENDING_REVIEW: subtle gray badge "Under Review"
  APPROVED: green badge "Profile Approved ✓"
  REVISION_REQUESTED: amber alert box (not just a badge) showing:
    "Updates requested by the Fincava team: [revisionNote]"
    CTA: "Update My Profile" scrolls to first incomplete or flagged section
  NEEDS_ATTENTION: amber badge "Flagged for Review" — no note shown to buyer
  Null / no profile: no badge shown

Data source: read p2ApprovalStatus and p2_revision_note from GET /api/buyer/onboarding response (add these two fields to the endpoint's SELECT).
