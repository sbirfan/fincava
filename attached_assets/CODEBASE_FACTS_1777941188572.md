## Auth System — CRITICAL

**JWT-based, NOT session-based.**

```typescript
// requireAuth sets on req:
(req as any).userId   // number — the logged-in user's ID
(req as any).userRole // string — "BUYER" | "SUPPLIER" | "ADMIN"

// NOT req.user — there is no req.user object
// NOT req.supplierId — does not exist
```

**Existing middleware:**
- `requireAuth` — in `lib/auth.ts` — verifies JWT cookie or Bearer token
- `requireAdmin` — in `middleware/admin.ts` — checks `(req as any).userRole === 'ADMIN'`
- `adminOnly` — `[requireAuth, requireAdmin]` as const — use this for admin routes
- `requireRole(role)` — async factory, rarely used
- **`requireSupplier` — DOES NOT EXIST. Must be created for CC-1D.**

## Supplier Identity — CRITICAL DESIGN GAP

**There are TWO supplier populations in this codebase. This is unresolved in the TDD/SOT.**

| Population | Table | Identity | Auth |
|------------|-------|----------|------|
| Farm suppliers | `suppliersTable` | `whatsappNumber` | **No user account linked** |
| Marketplace suppliers | `usersTable` (role=SUPPLIER) | email + password | JWT via `requireAuth` |

`suppliersTable` has **no `userId` FK**. Farm suppliers onboarded via WhatsApp have no corresponding row in `usersTable`. The existing `supplier-dashboard/` pages use `useGetSupplierStats()` from the API client — this is for marketplace suppliers (usersTable role=SUPPLIER).

**This means:** CC-1D compliance flows are targeted at farm suppliers (WhatsApp-onboarded). They currently have no login mechanism. This is an open architecture decision that must be resolved before CC-1D can be built. Options: (a) create user accounts for farm suppliers at onboarding, (b) use WhatsApp OTP auth, (c) officer-mediated flow (no supplier auth at all for Phase I).

### Confirmed column names (critical for SQL)
| Concept | Actual column | Actual enum values |
|---------|--------------|-------------------|
| Graduation state | `sellable_status` / `sellableStatus` | NOT_READY, ELIGIBLE, SELLABLE, PUBLISHED |
| AI quality score | `commercial_score` / `commercialScore` | integer 0–100 |
| Supplier name | `nombre_completo` / `nombreCompleto` | text |
| Supplier type | `supplier_type` / `supplierType` | FARMER, COOPERATIVE, EXPORTER |
| Compliance booleans | `rut_dian`, `ica_registro`, `fitosanitario_cert`, `dian_exportador` | boolean |

### Two-track model (actual implementation)
The "two-track entity model" in the TDD is implemented as a **single `supplierType` enum on `suppliersTable`**, not as separate entity tables:
- Track 1 (FARMER) → `supplierType = 'FARMER'`
- Track 1 (COOPERATIVE) → `supplierType = 'COOPERATIVE'`
- Track 2 (EXPORTER) → `supplierType = 'EXPORTER'`

## Migration Pattern

**NOT inline DDL in startup.** `index.ts` has NO DDL. Schema is managed via drizzle-kit.

```bash
# Correct migration workflow for new CC-1 tables:
# 1. Add schema to lib/db/src/schema/compliance-concierge.ts
# 2. Export from lib/db/src/schema/index.ts
# 3. Push to DB (in Replit terminal, from lib/db/):
pnpm --filter @workspace/db push
# OR generate + run migration:
pnpm --filter @workspace/db generate   # creates lib/db/drizzle/ migration file
pnpm --filter @workspace/db migrate    # runs lib/db/src/migrate.ts
```

**Migration files** (drizzle-kit generated) live in `lib/db/drizzle/` (auto-created by drizzle-kit).

## CC-1A Status — CONFIRMED NOT BUILT

The graduation service (`supplier-graduation-service.ts`) reads `complianceDocsTable` booleans directly. `aiOutputsTable.complianceGaps` is written by the AI scoring call but is **never parsed back into `complianceDocsTable` booleans**. The write-back hook described in TDD §2.14 is not implemented.

**This must be built before CC-1B makes sense architecturally.** The additive compliance layer builds on top of the boolean layer — if the boolean layer isn't being updated by AI scoring, the existing `complianceDocsTable` values are stale user input, not AI-verified signals.
